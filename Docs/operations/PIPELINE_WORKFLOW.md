# Content Pipeline Workflow

Date: 2026-07-04
Status: Baseline
Source: `tools/build_published_content.py`, `tools/extract_taxonomy_registry.py`

## End-to-End Flow

```
Workbooks → Extract → Merge → Enrich → Build → Shard → Index → Report → Deploy
```

This document traces the full operator workflow from workbook update through production deploy, including failure modes and recovery steps at each stage.

---

## Stage 1: Generate Taxonomy Registry (Optional, One-Time Per Editorial Pass)

The taxonomy registry (`data/taxonomy-registry.json`) holds curated editorial classifications. If terms need to be added or reclassified, run the extract script first:

```bash
.venv/bin/python3 tools/extract_taxonomy_registry.py \
  --glossary-workbook data_glossary.xlsx \
  --structure-workbook data_structure.xlsx \
  --out data/taxonomy-registry.json
```

This scans the workbooks and extracts all existing taxonomy mappings into a human-editable JSON file. The operator can then open `data/taxonomy-registry.json` and add/edit category entries for specific terms.

**When to do this:** Adding editorial classifications for named entities (datasets, tools, vendors) that auto-classification rules cannot handle.

### Command

```bash
.venv/bin/python3 tools/build_published_content.py \
  --glossary-workbook data_glossary.xlsx \
  --structure-workbook data_structure.xlsx \

Expected output:

```

Expected runtime: ~3.5 minutes on a modern laptop.

### CLI Flags

| Flag | Purpose | When to Use |
|---|---|---|
| `--coverage-threshold 0` | Disable the coverage gate | During development, testing, or iterating on rules |
| `--taxonomy-registry __disabled_registry_for_fixture__` | Disable editorial registry | Running tests or clean-room analysis |
| `--taxonomy-registry /path/to/custom.json` | Use a custom registry | Testing editorial changes before merging |

## Stage 3: Verify

### 3.1 Coverage Check

Read `public/content/published/reports/content-audit.json`:

```bash
python3 -c "import json; a=json.load(open('public/content/published/reports/content-audit.json')); print(f'Coverage: {a[\"coverage\"][\"taxonomyCoverageRatio\"]:.2%}, Issues: {a[\"qualityChecks\"][\"highSeverityIssueCount\"]}')"
```

**Pass criteria:**
- `taxonomyCoverageRatio` ≥ 0.74 (or whatever threshold is appropriate)
- `highSeverityIssueCount` = 0

**Fail recovery:** If coverage dropped or there are high-severity issues:
1. Check if the taxonomy registry was loaded correctly
2. Check if auto-classification rules need updating
3. Check if workbook columns changed
4. Re-run with `--coverage-threshold 0` to see warnings

### 3.2 Determinism Check (Recommended Before Deploy)

```bash
cp -r public/content/published /tmp/pre-deploy-check
.venv/bin/python3 tools/build_published_content.py \
  --glossary-workbook data_glossary.xlsx \
  --structure-workbook data_structure.xlsx \
  --out-dir /tmp/second-build --coverage-threshold 0
diff -r public/content/published /tmp/second-build | head -50
```

**Pass criteria:** No output from `diff`.

**Fail recovery:**
- If outputs differ, the build is non-deterministic. Common causes:
  - Unordered dict/set iteration in the build script
  - Timestamp embedded in output
  - Workbook file changed between runs

### 3.3 Path Count Check

```bash
python3 -c "import json; print(f'Paths: {json.load(open(\"public/content/published/paths/index.json\"))[\"pathCount\"]}')"
```

**Pass criteria:** `pathCount` should be ≥ the previous value. A decrease means learning paths were lost.

## Stage 5: Add Auto-Classification Rules (Optional)

When new unclassified terms need to be covered by keyword rules:

1. Analyze unclassified terms: examine `reports/content-audit.json` → `coverage.unclassifiedTerms`
2. Identify high-frequency tokens: see `Docs/data/UNCLASSIFIED_TERM_ANALYSIS.md`
3. Add rules to the appropriate batch list in `tools/build_published_content.py`:
   - `BATCH_5` for data/processing patterns (p8)
   - `BATCH_6` for acronyms/concepts (p7-9)
   - Or directly in `AUTO_CLASSIFICATION_RULES` for one-off rules
4. Run syntax check: `python3 -c "import py_compile; py_compile.compile('tools/build_published_content.py', doraise=True)"`
5. Rebuild with `--coverage-threshold 0`
6. Verify coverage delta: `python3 -c "print(f'Delta: {new_coverage - old_coverage:.2%}')"`
7. Spot-check 10-20 classified terms per new rule for correctness
8. Run tests: `.venv/bin/python3 -m unittest tests.test_build_published_content -v`
9. Commit: `git add tools/build_published_content.py public/content/published/ && git commit -m "Add Batch N auto-classification rules"`

## Failure Mode Recovery

### Build Fails with "Workbook not found"

```bash
ls -la data_glossary.xlsx data_structure.xlsx
```

If missing, check:
- Are you in the repo root? (`pwd`)
- Did the file get moved? Check `git status` for renamed files
- Is the file on a mounted volume? Check `df` and permissions

### Build Fails with Memory Error

```bash
# Increase Python memory limit (macOS)
export PYTHONMALLOC=malloc
ulimit -S -m unlimited
```

If still failing, the term count may have exceeded the script's capacity (~600 MB peak for 18k terms).

### Build Produces Lower Coverage Than Expected

```bash
# Compare against the last known good build
git diff HEAD -- public/content/published/reports/content-audit.json
```

Check:
- Did the workbook change? (`git diff -- data_glossary.xlsx`)
- Did auto-classification rules get removed? (`git diff -- tools/build_published_content.py`)
- Did the taxonomy registry get disabled? (Check the CLI command for `__disabled_registry_for_fixture__`)

### Published Artifacts Are Stale in Git

If `public/content/published/` in git doesn't match the current build:

```bash
# Rebuild to the same output directory
.venv/bin/python3 tools/build_published_content.py --glossary-workbook data_glossary.xlsx \
  --structure-workbook data_structure.xlsx --out-dir public/content/published

# Verify consistency
python3 -c "
import json
a = json.load(open('public/content/published/reports/content-audit.json'))
print(f'Terms: {a[\"termCount\"]}, Coverage: {a[\"coverage\"][\"taxonomyCoverageRatio\"]:.2%}')
"
```

### Worker Deploy Fails

```bash
# Check wrangler configuration
npx wrangler whoami
npx wrangler deploy --dry-run

# Common issues:
# - Missing D1 binding: check wrangler.jsonc
# - Missing environment variables: check .dev.vars or Cloudflare dashboard
# - Build errors: npm run build first
```

### Debugging Commands

```bash
# Syntax check the build script
python3 -c "import py_compile; py_compile.compile('tools/build_published_content.py', doraise=True)"

# Quick coverage delta vs previous build
python3 -c "
import json
old = json.load(open('/tmp/previous-build/reports/content-audit.json'))
new = json.load(open('public/content/published/reports/content-audit.json'))
print(f'Coverage: {old[\"coverage\"][\"taxonomyCoverageRatio\"]:.2%} -> {new[\"coverage\"][\"taxonomyCoverageRatio\"]:.2%}')
print(f'Unclassified: {old[\"coverage\"][\"unclassifiedTerms\"]} -> {new[\"coverage\"][\"unclassifiedTerms\"]}')
"

# Check if any term appears in multiple shards
python3 -c "
import json, os, re
slugs = []
for f in os.listdir('public/content/published/terms/shards'):
    if f.endswith('.json') and f != 'shard-manifest.json':
        shard = json.load(open(f'public/content/published/terms/shards/{f}'))
        slugs.extend(t['slug'] for t in shard)
seen = set()
for s in slugs:
    if s in seen: print(f'DUPLICATE: {s}')
    seen.add(s)
print(f'Total: {len(slugs)}, Unique: {len(seen)}')
"
```
